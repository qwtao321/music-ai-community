# AI 音乐社区 P0 需求与模块现状清单

## 文档说明
本文档用于梳理当前 AI 音乐社区项目的开发进展，明确：

1. 当前哪些模块已经真实接入后端
2. 哪些模块仍处于只有界面、半接入或伪接入状态
3. 面向“对外公测上线”的后续 P0 功能需求清单

当前判断口径按“公测上线”标准，不按单纯演示版本标准定义。

---

## 一、当前已真实接入后端的模块

### 1. 音乐生成主流程（原创）
已具备基础后端闭环：

- 前端通过 `/api/generate` 提交生成请求
- 后端创建 `generation job`
- 通过 Suno-like provider 发起真实生成任务
- 前端通过 `/api/jobs/:id` 轮询生成状态
- 成功后将结果保存为草稿作品（track）
- 草稿可后续发布到广场

### 2. AI 写歌词
已接入真实后端能力：

- 路由：`/api/lyrics`
- 模型：DeepSeek
- 已有独立 prompt 组装逻辑
- 支持根据主题、语言、标签、歌名生成歌词

### 3. AI 生成歌名
已接入真实后端能力：

- 路由：`/api/song-title`
- 模型：DeepSeek
- 已有独立歌名 prompt 逻辑
- 支持根据提示词、语言、标签生成可发行歌名

### 4. 音乐广场
已具备真实后端数据读取能力：

- 路由：`/api/hall`
- 支持作品列表读取
- 支持搜索、分类、排序
- 页面 `/music/hall` 已接入后端数据

### 5. 音乐榜单
已具备真实后端能力：

- 路由：`/api/rankings`
- 支持热度榜、播放榜、点赞榜、收藏榜、同款榜
- 页面 `/music/rankings` 已接入后端数据

### 6. 歌曲详情页与播放
已具备基本后端能力：

- 详情页 `/music/[id]` 可读取真实作品数据
- 播放接口 `/api/tracks/:id/play` 已存在
- 前端播放动作会回写播放次数

### 7. 点赞 / 收藏
已具备真实后端能力：

- `/api/tracks/:id/like`
- `/api/tracks/:id/favorite`
- 已可写回作品互动数据

### 8. 我的页面
已具备真实后端读取能力：

- 路由：`/api/me`
- 支持读取：
  - 我的作品
  - 我的收藏
  - 我的生成任务
  - 积分流水

### 9. 作品发布
已具备真实后端能力：

- `/api/jobs/:id/publish`
- `/api/tracks/:id/publish`
- 支持草稿发布为公开作品
- 支持草稿、公开、隐藏三种可见性状态

### 10. 管理后台基础接口
已具备最小后端能力：

- `/api/admin`
- `/api/admin/credits`
- `/api/admin/hide-track`

可支持：

- 查看管理快照
- 发放积分
- 隐藏作品

### 11. 模板库数据
模板库不只是前端静态展示：

- store 已支持 `listStyleTemplates()`
- CloudBase store 中也有模板读取/初始化逻辑
- 创作页模板数据来自后端数据源能力，而非纯前端常量

### 12. CloudBase 数据存储能力
项目已经有 CloudBase 持久化实现，涉及：

- profiles
- generation jobs
- tracks
- credit ledger
- likes
- favorites
- provider assets
- style templates

## 二、当前只有界面、半接入或伪接入的模块

这一部分是当前最需要重点关注的，因为用户会看到入口，但实际能力并未完整闭环。

### 1. 音频翻唱上传
当前状态：**只有界面，未完成真实后端接入**

现状：

- 创作页已有“上传参考音频”按钮
- 选择文件后，前端只是拼接了一个假的本地 URL
- 没有真实上传到 CloudBase Storage
- 没有上传进度、失败处理、格式校验、大小校验、时长校验

结论：

- 当前“上传参考音频”是界面存在，但未形成真实上传链路

### 2. `cover_audio` 音频翻唱模式
当前状态：**后端字段存在，但未真实打通生成能力**

现状：

- schema 中已有 `referenceAudioUrl`
- store 和 generation job 中也会保存该字段
- 但当前 provider 构造 Suno 请求时，并未真正把参考音频用于生成

结论：

- 当前“音频翻唱模式”在产品层已存在
- 但在真实生成层面还没有形成实际翻唱能力

### 3. `cover_text_style` 风格重制模式
当前状态：**模式存在，但后端无独立实现策略**

现状：

- 前端有模式入口
- 后端 job / track 也支持该 mode
- 但 provider 没有针对该模式做独立参数映射、请求策略、结果处理

结论：

- 目前更像“模式名存在”
- 还不是一项真实独立的生成能力

### 4. 登录体系
当前状态：**模块存在，但主产品运行时未正式启用**

现状：

- 有登录页
- 有短信验证码登录面板
- 有 CloudBase session 同步接口
- 但当前线上产品为了绕开问题，主流程已切为匿名体验模式
- 登录入口已被隐藏
- 主流程不依赖真实登录即可使用

结论：

- 当前不是“登录功能已完成”
- 而是“登录模块存在，但产品正式运行在匿名降级模式”

### 5. 退出登录
当前状态：**组件存在，但未作为正式产品能力暴露**

现状：

- 退出登录组件仍在代码内
- 当前导航中未作为正式能力开放

结论：

- 会话管理链路未按正式产品标准完成

### 6. 生产环境真实数据保障
当前状态：**CloudBase 已接入，但存在 demo 回退逻辑**

现状：

- 为避免线上 500，当前 server store 增加了 fallback
- 当 CloudBase 初始化失败或方法调用失败时，会回退到 demo store

结论：

- 这有助于演示可用性
- 但不满足公测标准
- 因为生产环境异常时，用户可能看到演示数据而非真实业务数据

### 7. 管理后台前端可操作性
当前状态：**后端已有，前端仍偏工程调试面板**

现状：

- 加积分和隐藏作品需要手填 `profileId`、`trackId`
- 缺少搜索、筛选、详情查看、误操作保护

结论：

- 不属于“没后端”
- 但属于“只有最低可用界面，未达到运营后台标准”

## 三、后续必须补齐的 P0 功能需求

以下 P0 清单按“对外公测上线”标准定义。

### P0-1. 恢复真实用户体系
目标：让用户身份、作品归属、互动行为都回到真实账号体系。

需求包括：

- 恢复登录入口可见
- 修复短信验证码登录闭环
- 恢复退出登录
- “我的”“发布”“点赞/收藏”“管理”切回真实身份校验
- 如果保留匿名模式，要明确为“游客模式”，不能与正式用户体系混用

### P0-2. 打通真实音频翻唱链路
目标：让“音频翻唱”成为真实可用功能，而不是伪入口。

需求包括：

- 上传参考音频到 CloudBase Storage
- 生成可持久访问的真实文件 URL
- 将参考音频真实传递给 Suno / Provider
- 记录 reference asset 元数据
- 增加上传校验：
  - 文件格式
  - 文件大小
  - 音频时长
  - 上传失败提示

### P0-3. 让风格重制变成真实能力
目标：让 `cover_text_style` 不只是一个 UI 模式。

需求包括：

- 定义独立输入策略
- 与原创 / 音频翻唱模式区分开
- 有独立 prompt 拼装逻辑
- 有清晰结果追踪方式
- 如果当前 provider 不支持真风格重制，则需改为明确的“同风格重写”方案，不应伪装为已实现

### P0-4. 补齐生成任务稳定性
目标：让生成链路在真实线上环境稳定可用。

需求包括：

- 服务端持续刷新任务状态，而不是只依赖前端轮询
- 支持 provider 延迟返回
- 支持失败重试 / 幂等处理
- 保证积分扣减和返还的一致性
- 保存 provider 原始错误信息，方便后台排查

### P0-5. 生产环境真实数据库保障
目标：生产环境必须优先真实数据，不应静默落回演示态。

需求包括：

- 明确生产环境强制使用 CloudBase
- demo fallback 仅保留给本地开发 / 演示环境
- 增加环境健康检查
- 防止线上展示 demo seed 数据
- 校验 tracks / jobs / ledger / assets / templates 的生产一致性

### P0-6. 补齐基础运营后台
目标：让后台能被实际运营或管理员使用，而不只是技术调试面板。

需求包括：

- 用户搜索
- 作品搜索
- 失败任务详情查看
- 草稿 / 隐藏作品查看
- 可视化加积分
- 基础防误操作能力

### P0-7. 补齐基础风控与合规提示
目标：降低翻唱与内容生成场景的上线风险。

需求包括：

- 音频翻唱版权 / 授权提示
- 提示词、歌词、上传内容最小校验
- 上传文件限制说明
- 失败提示与用户可理解的错误反馈
- 最小免责声明

### P0-8. 补齐线上可观测性与配置校验
目标：让问题能定位，而不是靠人工猜测。

需求包括：

- Suno key 检查
- DeepSeek key 检查
- CloudBase env 检查
- Storage 能力检查
- 短信能力检查
- 基础链路监测：
  - 生成失败率
  - 发布失败率
  - 上传失败率
  - 登录失败率

## 四、建议的模块状态分级

为了后续产品管理和开发排期，建议将当前模块状态分为 3 档。

### A. 已真实接入后端
包括：

- 原创生成
- AI 写歌词
- AI 生成歌名
- 草稿生成与发布
- 音乐广场
- 榜单
- 我的页面
- 播放 / 点赞 / 收藏
- 基础管理接口
- 模板库数据能力
- CloudBase 数据模型

### B. 半接入 / 伪接入
包括：

- 音频翻唱上传
- `cover_audio` 模式真实生成
- `cover_text_style` 真实生成
- 登录体系
- 退出登录
- 生产环境真实数据保障
- 管理后台可操作性

### C. 公测 P0 必补
包括：

- 真实登录体系
- 真实音频翻唱链路
- 真实风格重制链路

- 生产数据可靠性
- 可运营后台
- 基础风控与合规
- 可观测性与配置检查

## 五、后续开发完成后的验收场景

以下场景建议作为后续 P0 完成后的验收标准。

### 1. 原创生成
- 用户登录后可生成原创歌曲
- 结果真实落库
- 可在“我的作品”中播放
- 可发布到广场

### 2. 音频翻唱
- 用户可上传真实音频文件
- 文件真实存储到 CloudBase Storage
- 生成任务可追踪 reference asset
- 结果可播放、可发布

### 3. 风格重制
- 与原创 / 音频翻唱形成明确能力区分
- 生成任务、结果作品、后台记录都可识别其模式来源

### 4. 社区可见性
- 草稿作品不会进入广场
- 发布后进入广场
- 隐藏作品不会出现在广场和榜单

### 5. 数据可靠性
- 生产环境异常时，不展示 demo seed 数据
- CloudBase 读写异常可被检测和告警

### 6. 后台可运营
- 可搜索用户和作品
- 可查看失败任务与错误原因
- 可发放积分与隐藏作品

### 7. 身份权限
- 游客、普通用户、管理员三类身份能力边界清晰
- 登录、退出、发布、管理权限全部符合预期

## 六、当前默认假设

为避免范围失控，当前清单默认基于以下假设：

- 当前按“对外公测上线”定义 P0，而不是演示版本
- 支付、套餐购买、评论、关注、私信暂不纳入 P0
- 当前积分体系先沿用“免费积分 + 管理员发放”
- Suno / 类 Suno provider 仍是音乐生成主后端，不改变核心架构
- 后续文档会以此清单为准继续拆详细排期与实现计划
